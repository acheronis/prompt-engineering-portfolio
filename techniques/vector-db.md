# Векторные базы данных и Chunking

## Суть
Векторная БД хранит текст в виде числовых векторов (embeddings).
Поиск идёт не по ключевым словам а по смысловой близости.

## Chunking — нарезка текста

Правило: один смысловой блок = один чанк.
Плохая нарезка → рваные ответы.
Хорошая нарезка → точный поиск.

Параметры:
- Chunk length: 200-500 символов для коротких фактов
- Chunk overlap: 20-50 символов чтобы не терять контекст на границах

## Методы индексации

High Quality — embedding модель, точный поиск, тратит токены.
Economical — ключевые слова, быстро, бесплатно, менее точно.

## Методы поиска

Векторный — ищет по смыслу. "Тёплая одежда" найдёт "куртка".
Полнотекстовый — ищет по словам. Точный но негибкий.
Гибридный — комбинирует оба. Лучший результат.

## Реализация в Dify

Создана Knowledge Base "Product Catalog":
- 4 товара → 4 чанка по одному товару каждый
- Chunk length: 200 символов
- Каждый чанк содержит: название, цвет, размеры, цену, материал

Плохой пример (PDF, chunk 500):
"ХудиоверсайзЦвет:чёрный" — слова слиплись, поиск не работает.

Хороший пример (TXT, chunk 200):
"Товар: Худи оверсайз. Цвет: чёрный, белый, серый." — чистый чанк.

## Ключевой вывод
Качество RAG системы на 50% зависит от качества chunking.
Плохо нарезал → даже лучшая модель даст плохие ответы.
