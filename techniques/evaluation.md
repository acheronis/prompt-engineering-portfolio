# Evaluation Framework

## Суть
Систематическая оценка качества промптов по рубрике.
Без оценки непонятно улучшился промпт или нет.
Без метрик невозможно доказать ценность работы.

## Три метода оценки

### 1. A/B сравнение
Два промпта, одна задача, сравниваешь результат по критериям.
Используй для: выбора между двумя подходами.

### 2. Рубрика (scorecard)
Задаёшь критерии и выставляешь баллы от 1 до 10.

Пример рубрики для поддержки клиентов:
- Точность информации: 1-10
- Тон и дружелюбность: 1-10
- Структура ответа: 1-10
- Actionability (есть ли чёткий следующий шаг): 1-10

### 3. LLM-as-a-Judge
Просишь модель оценить два ответа по твоей рубрике.
Реальная техника применяемая в production AI-системах.

## Пример из практики

**Задача:** оценить два промпта для ответа на вопрос о наборе веса.

| Критерий | Промпт A (без CoT) | Промпт B (с CoT) |
|---|---|---|
| Точность | 5/10 | 5/10 |
| Структура | 4/10 | 5/10 |
| Практичность | 5/10 | 5/10 |
| Ясность | 4/10 | 5/10 |
| **Итого** | **18/20** | **20/20** |

Оценка модели: Промпт A — 17/20. Найдено внутреннее противоречие:
"одна причина" в первой фразе, но 5 причин в тексте.
Моя оценка это пропустила. Модель поймала.

## Правила чистого тестирования
- Всегда тестируй в чистой сессии — контекст предыдущих ответов искажает baseline
- Фиксируй версию промпта и результат — без этого итерации бессмысленны
- Оценивай по одним критериям для всех версий — иначе сравнение нечестное

## Ключевой вывод
LLM-as-a-Judge замечает противоречия которые человек пропускает.
Комбинация своей оценки + оценки модели даёт наиболее точный результат.
Метрики превращают субъективное "лучше/хуже" в измеримый результат.
